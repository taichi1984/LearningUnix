# 研究課題10.02
## 課題内容
pipe.c では、親プロセスはデータを消費するプログラム、子プロセスはデータを生産するプログラムを実行する。
役割を逆転させるとどのような違いが現れるか。if(pid > 0)というテストを if(pid == 0)に書き換えると、役割が逆転する。
何が起きるのか、それはなぜかを答えなさい。

## pipe.c
### ソースコード

```

/* pipe.c
 * 目的：１つのプロセスから別のプロセスにデータを流す。
 * パイプラインの作り方を示す。引数として２つのコマンドを取り、av[1]の出力をav[2]の出力に接続する。
 *
 *  使い方 pipe command1 command2
 *  効果 : command 1 | command2
 *  制限 : コマンドは引数を取ることができない。
 *  引数の個数がわかっているのでexeclp()を使う。
 *  注意：子と親を取り替えると面白いことになる。
 */

#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>

#define oops(m, x)                                                             \
  {                                                                            \
    perror(m);                                                                 \
    exit(x);                                                                   \
  }

int main(int ac, char **av) {
  int thepipe[2], /*2つのファイルディスクリプタ*/
      newfd, pid;

  if (ac != 3) {
    fprintf(stderr, "usage:pipe cmd1 cmd2\n");
    exit(1);
  }
  if (pipe(thepipe) == -1)
    oops("Cannot get a pipe", 1);

  /* ----------------------------------------------------------
   * パイプを作ったのでプロセスを２つにする */

  if ((pid = fork()) == -1)
    oops("Cannot fork", 2);

  /* -----------------------------------------------------------
   * この時点でプロセスは２個ある。親はパイプから読み出す*/

  if (pid > 0) {       /*親はav[2]を実行する*/
    close(thepipe[1]); /*親はパイプに書き込まない*/

    if (dup2(thepipe[0], 0) == -1)
      oops("could not redirect stdin", 3);

    close(thepipe[0]); /*標準入力の複製を作り、パイプをクローズする。*/
    execlp(av[2], av[2], NULL);
    oops(av[2], 4);
  }

  /* 子はav[1]を実行し、パイプに書き込む*/
  
    close(thepipe[0]); /*子はパイプから読み出さない*/

    if (dup2(thepipe[1], 1) == -1)
      oops("could not redirect stdout", 4);

    close(thepipe[1]); /*標準出力の複製を作り、パイプをクローズする。*/

    execlp(av[1], av[1], NULL);
    oops(av[1], 5);
  
}

```

### 実行結果
```
$ ./pipe ls sort
10.02.md
pipe
pipe.c
```

並び替えが行われる。

## ソースコードの親と子を書き換える。
if(pid > 0)の条件を if(pid == 0)に書き換え、もとの子プロセスの処理を親プロセスにさせて、親プロセスの処理を子プロセスにさせる。

```

/* pipe.c
 * 目的：１つのプロセスから別のプロセスにデータを流す。
 * パイプラインの作り方を示す。引数として２つのコマンドを取り、av[1]の出力をav[2]の出力に接続する。

 *  効果 : command 1 | command2

 *  引数の個数がわかっているのでexeclp()を使う。
 *  注意：子と親を取り替えると面白いことになる。
 */

#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>

#define oops(m, x)                                                             \
  {                                                                            \
    perror(m);                                                                 \

  }

int main(int ac, char **av) {
  int thepipe[2], /*2つのファイルディスクリプタ*/

  /* -----------------------------------------------------------

  if (pid == 0) {       /*子はav[2]を実行する*/

```

### 実行結果

```

$ ./pipe2 who sort
taichi84 :1           2026-03-03 23:10 (:1)
taichi84 seat0        2026-03-03 23:10 (login screen)
$ ./pipe2 who sort
$ taichi84 :1           2026-03-03 23:10 (:1)
taichi84 seat0        2026-03-03 23:10 (login screen)
```

新たにpipe2プログラムを作ってwhoで試した。
順番的にはきちんとソートされ問題ないように思われるが、何回か実行するとまれに次のプロンプトが表示されてから pipeの内容が出力されることがあるようだ。
10回程度の試行回数ではあるが、pipeでは発生しないが、pipe2のみ発生することがあった。

## 何が起こっているのか？
基本的にpipeの機能は両方正しく動いており、パイプを通して、１つ目のプロセスの出力を２つ目のプロセスの入力に送ることには成功している。
親プロセスの役割と子プロセスの役割が逆転しても、データを送る事自体は問題ないようだ。

親が読み出すケース、つまり親がパイプから入力を受けとり、画面に表示すると
なぜ次のプロンプトが表示されてからpipeの内容が出力されることがあるのだろうか？

### シェルとforkの関係を整理する。
シェルはなにかプログラムを実行するときに、fork → execをすることによって指定されたプログラムを走らせる。
そしてその場合、シェルは親プロセス、実行されたプロセスは子プロセスの関係になり、シェルは子プロセスの終了をwaitする。

実行されたプロセス（今回はpipe2)がforkして処理を行った場合、pipe2がforkしたプロセスはシェルから見ると子プロセスの子プロセスとなる。
シェルは自らの子プロセスの終了に関してはwaitするが、子プロセスの子プロセスはwaitしない。

そしてforkされたプロセスの終了順序はスケジューラに依存し、保証されていない。

よって、pipe2の場合、子がpipeから読み出して、文字を出力する設計にした場合、先にpipe2の親プロセスが終了した場合に、シェルのwaitが完了して先にプロンプトを出力してから
子プロセス（シェルからみたら孫プロセス）がpipeからの読み出しを完了して、画面に出力することが起こりうる。

pipeの親プロセスに表示する役割をもたせた場合、プロンプトを表示するシェルのプロセスは必ずシェルの子プロセスであるpipeの親プロセスが表示したのを待ってからプロンプトを表示する。
そのため、プロンプトが先に表示されてしまい、画面表示がおかしくなることは起こらない。

## 結論
pipeの親プロセスと子プロセスの役割を逆にした場合、
プロンプトの再表示とpipe2の表示の出力順が逆になってしまうことがあるのは、シェルがどのプロセスを待っているのかが関係する。
親プロセスに表示をさせないと、子プロセスが先に処理を終えた場合、表示を待たずにシェルが先にプロンプトを表示してしまうことがある。
