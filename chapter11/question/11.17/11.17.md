# プログラミング課題11.17
## 課題内容
**キャッシュとプロクシについてさらに** 前問のキャッシングプロキシは、馬鹿げた考え方だ。
fingerサーバのために、キャッシュを使ったときに意味があるのはなぜか説明しなさい。
また、ユーザー情報のキャッシュを管理するfingerサーバーを書きなさい。
時報サーバのキャッシュの項目には、自然な寿命があるが、fingerサーバーのキャッシュでユーザー情報をどのくらい残しておくかはどのように決めればよいだろうか。

## fingerサーバーのためにキャッシュを使ったときに意味があるのかはなぜか説明しなさい。
### そもそも時報サーバーのプロキシはなぜ馬鹿げた考え方なのか
キャッシュはそもそも「欲しいデータは更新されていない可能性が高い」ことを前提にキャッシュをして、
元データを再取得するために通信することなく、情報を返すものだ。
時報サーバーは当然のことながら、1秒ごとに必ず更新されるものなので、プロキシを使うのは馬鹿げている。

### ではfingerサーバーはどうなのか。
fingerサーバーは個人情報検索サーバーである。
個人情報が更新されるタイミングは電話番号が変わったり引っ越しして住所が変わったり、
登録されているユーザーが追加、削除されたりするタイミングである。毎日変わるようなものでもないし、ましてや毎秒かわることでもない。
今回自分の作ったfingerサーバーのケースでも

ユーザー名
ユーザーID
グループID
ホームディレクトリ
シェル

といった内容のため、一般的なケースを考えれば1日の間に何度も変わるようなこともなく、変更されるとしても１日以上の特定の期間の間に１回程度のものだろう。
ただ、ある人にコンタクトをするために個人情報を調べたい人は１日に複数人いる可能性は十分にある。
そのため、fingerサーバーのプロキシでは問い合わせ情報があったら、キャッシュを持っておくのは十分意味があることと言える。



## fingerサーバーではユーザー情報をどのくらい残しておけばいいのだろうか？
これは要件とシステムの構成にもよるが、基本的にはfingerサーバーはプロキシサーバーに変更通知をしないものと考えた場合、
このfingerサーバーの主目的に依存するものと思われる。
基本的には、住所の変更（引っ越し）が行われたり、電話番号の変更が行われるのは１日単位の話になるので、
１日に１回情報を取れば十分と思われる。
今回のfingerサーバーでは以上の理由から１日に１回とするが、実際どれぐらいの期間キャッシュを持っておくべきかという問題は、
作りたいものの業務用件に依存するものになるだろう。

## ユーザーの情報をどうキャッシュしておけばいいだろうか？
fingerサーバーは検索された人の個人情報を返す。
proxyサーバーは基本的にはクライアントから受け取ったリクエストをそのままfingerサーバーに流すだけだが
キャッシュを持っておくためには、クライアントから受け取ったリクエストのリストとそのレスポンスのリストをもっておかなければならないだろう。
時報サーバーと違い、多様なリクエストが来る可能性があるので、そこは考慮に入れなければならない。
どのようにリクエストとレスポンスを保持すればいいだろうか？


### どのように保存し、どのように扱うか。
考えられる最も簡単な方法はHashMapのような構造に保存することである。
リクエストをキー、レスポンスをバリューにして保存する。
リクエストが来たら、そのHashMapからキーを検索し、一致した場合はそのレスポンスを返す。
一致しなかった場合は、その情報を本物のfingerサーバーに取得しにいき、取得したものをレスポンスで返す。
レスポンスが正しく返ってきた場合に、そのリクエストとレスポンスの内容をHashMapに追加すればよい。（失敗時には追加の必要はない）

キャッシュを削除するタイミングはどうすべきだろうか？
考えられるのは、HashMapのサイズを超えたとき、ある特定の時間を超えたとき、更新を検知したときとなる。
今回は特に更新は通知されないので、１日１回ある特定の時間を超えたときにHashMapの内容を全て消せばよいことにする。
HashMapのサイズは今回はどれぐらいのアクセスがあるかの要件が具体的にあるわけではないので、
動的サイズが変わるようにして、１日分はずっと貯まるようにしたい。(今回は課題なので悪意のある過剰アクセスなどは考えない）
時間のみでキャッシュのクリアを行うものとする。

int setup_cliant(char *ipaddr, int port)
/*
 * クライアントとして第１引数で入力したサーバーに接続する。
 * 戻り値:成功時はsock_idを、失敗時は-1を返す
 */
{

  struct sockaddr_in servadd;
  char hostname[HOSTLEN];
  struct hostent *hp;
  int sock_id = -1;

  /* ステップ１：　ソケットを作成する*/
  if ((sock_id = socket(PF_INET, SOCK_STREAM, 0)) == -1) {
    return sock_id;
  }

  /** ステップ2 : サーバーに接続する*/

  memset(&servadd, 0, sizeof(servadd));
  hp = gethostbyname(ipaddr);
  if (hp == NULL) {
    perror("gethostbyname");
    return -1;
  }

  memcpy((struct sockaddr *)&servadd.sin_addr, hp->h_addr, hp->h_length);
  servadd.sin_port = htons(port);
  servadd.sin_family = AF_INET;

  if (connect(sock_id, (struct sockaddr *)&servadd, sizeof(servadd)) != 0) {
    perror("connect");
    return -1;
  }

  return sock_id;
}

int setup_server(int port) {
  struct sockaddr_in saddr;
  char hostname[HOSTLEN];
  int sock_id;

  /* ステップ1:カーネルにソケットを要求する。*/

  sock_id = socket(PF_INET, SOCK_STREAM, 0); // ソケット取得
  if (sock_id == -1)
    return -1;

  /* ステップ２：ソケットにアドレスをバインドする。アドレスはホスト、ポート*/

  memset((void *)&saddr, 0, sizeof(saddr)); /*構造体をクリアする*/

  saddr.sin_addr.s_addr = htonl(INADDR_ANY);
  saddr.sin_port = htons(
      PORTNUM); // htons()関数でリトルエンディアン→ビッグエンディアンに変換する。（ネットワークはビッグエンディアンが共通語になっている。）
  saddr.sin_family = AF_INET;

  if (bind(sock_id, (struct sockaddr *)&saddr, sizeof(saddr)) != 0)
    return -1;

  /*
   * ステップ3:キューサイズを１としてソケットを着信できる状態にする。
   */

  if (listen(sock_id, 5) != 0)
    return -1;

  return sock_id;
}

ssize_t write_all(int sock_fd, char *buf, int rn) {
  int wr = -1;
  int total = 0;
  while (total < rn) {
    wr = write(sock_fd, buf + total, rn - total);

    if (wr <= 0) {
      return -1;
    }
    total += wr;
  }
  return total;
}

int put_node(Node **table, int size, const char *key, const char *value)
/*
 *  第１引数は格納するMapをさすポインタ
 *  第２引数は追加したいNode
 *
 *  戻り値は、成功時は1
 *  エラー時は -1を返す。
 */
{
  int index = hash(key) % size;

  Node *tmp = table[index];
  while (tmp != NULL) {
    if (strcmp(tmp->key, key) == 0) {
      free(tmp->value);
      tmp->value = strdup(value);

      return 1;
    }
    tmp = tmp->next;
  }
  Node *node = malloc(sizeof(Node));
  if (!node)
    return -1;

  node->key = strdup(key);
  node->value = strdup(value);
  if (!node->key || !node->value) {
    free(node->key);
    free(node->value); free(node);
    return -1;
  }

  node->next = table[index];
  table[index] = node;

  return 1;
}

Node *get_node(Node **table, int size, char *key)
/*
 *  第１引数は取得するnodeが格納されたMapをさすポインタ
 *  第２引数は欲しいnodeのsize
 *  第３引数は欲しいnodeのkey
 *
 *  戻り値は欲しいNodeを返す。
 *  失敗時（エラー時、もしくは該当するNodeがない場合)はNULLを返す。
 */
{
  int index = 0;
  index = hash(key) % size;

  Node *tmp = table[index];
  while (tmp != NULL) {
    if (strcmp(tmp->key, key) == 0) {
      return tmp;
    }
    tmp = tmp->next;
  }
  return NULL;
}

void free_map(Node **map, int size) {
  for (int i = 0; i < size; i++) {
    Node *tmp = map[i];
    while (tmp != NULL) {
      Node *next = tmp->next;

      free(tmp->key);
      free(tmp->value);
      free(tmp);

      tmp = next;
    }
    map[i] = NULL;
  }
}

unsigned long hash(const char *str) {
  unsigned long h = 5381;
  int c;

  while ((c = *str++))
    h = h * 33 + c;

  return h;
}

## まとめ
HashMapを使うことで、requestとresponseのセットをうまく保存することができた。
キャッシュの仕組みは前の課題と同じ仕組みなので省略する。
なお今回は、簡略化のため各レコードはBUFSIZの中で収まるという前提で書いているが、本来ならreadやwriteはそれを超えても大丈夫なように対処しておくべきである。

データ構造はデータと使用用途にあわせて適切なものを使う必要がある。
今回はkeyがユーザーネームのため、そのままHashMapのkeyにできることと
Hashmapは平均の計算量がO(1)のため、高頻度のアクセスにも適していたため採用した。
