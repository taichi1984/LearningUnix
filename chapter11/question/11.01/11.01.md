# 研究課題 11.01
## 課題内容
**宅配ピザ注文プロトコル** 時報、電話番号検索サービスではなく、宅配ピザサービスを経営しているとすると、どうなるだろうか？
プロトコルはもっと複雑になるはずだ。宅配ピザサービスのサーバとクライアント間でやり取りされるメッセージシーケンスを説明しなさい。
なお、このプロトコルには、クライアントが１つの注文に複数の項目を追加できるようにするループが含まれることに注意しなさい。

### この話の前提
まず、サーバーにつながるまで、クライアントはメニューを知らないものとする。

## 宅配ピザサービスの仕様を考える　第１案（実際のピザ屋に寄せた案）
### 目的
クライアントはピザを注文したく、サーバーはピザの注文を受けた上で注文を表示し、OKかどうかをクライアントに問い合わせたい。
クライアントはそれでいいならば、yesの返答、悪いならばnoの返答をする。
それを受けてサーバーはyesならば、注文を受け付けた旨をサーバー側に伝える。noならば、注文のしなおしを行えれば良い。

### サーバー側のaccept後の動き
サーバーはacceptしたら、ピザ屋である旨を伝えるメッセージをクライアントに送る。
そして注文メニューを表示し、クライアント側からの注文を待つ。
クライアントから送られた注文内容をそのまま確認のため送り返し、OKかNGかのダイアログを出すようにクライアントに送る。


### クライアント側のconnect後の動き
クライアントは注文のメニューが表示されたら、クライアントに注文を行う。
任意の種類のピザを任意の枚数を入力し、サーバーに送信する。
サーバーの確認メッセージのデータをうけとったら、OKかNGかの情報を送る。
OKであることを送信したら、クライアントを終了する。
NGであることを送信した場合、再度注文受け付けを待ち、注文。
OKを選択するまでこれを繰り返す。

### 気になる点
TCPによる接続は送信したものの確認が行われるため、送信元と送信先のデータが相違ないものになることが保証されている。
ならば、クライアント側の注文をサーバーに送り、サーバーが注文があってるかの確認を聞き直す必要はあるのだろうか？
サーバーはメニューをクライアントに送り、注文を受けるだけでよいのではないだろうか？

クライアント側は最初にサーバーからもらったメニューをうけて、メニューリストを作ってクライアント側で表示する必要がある。
それをもとに注文用文章を作って、サーバーに送れば良い。クライアントプログラムを操作しているユーザーに対しての確認は、
送信前の時点で行えばよく、注文をOKするまでの一連の確認の流れはクライアントだけで完結できるのではないか？
そうすれば通信の回数も減るので、より効率の良いプログラムになりそうだ。

## 宅配ピザサービスの仕様を考える　（第２案）
### サーバー側
1,サーバー側はaccept後、クライアントに注文を受け付ける旨のメッセージとメニューリストのメッセージを送る。
2,その後、注文のリストを受け取ったら、注文を正常に受け付けたことをクライアントに通知し、受付終了。

### クライアント側
1,クライアント側はサーバーに接続したら、まずメニューのリストがサーバーからのメッセージにより表示される。
2,そのメニューを元にユーザーに注文させるインターフェースを作り、注文を受け付ける。
3,インターフェース上で注文の確定をOKされたら、注文リストを表示し、ユーザーに最終確認をさせる。
4,最終確認がOKの場合、その注文データをもとに注文用のメッセージを作り、サーバーに送る。
　最終確認がNGの場合、2に戻り、再度注文を受け付ける。
5,サーバーから注文受け付けた旨のメッセージを受け取り、終了する。

### この案のいいところ
サーバー側の仕事がメニューのメッセージを送り、注文を受け付け、注文受け付け済みのメッセージをを送るだけと第１案よりシンプルになる。
注文内容の確認にいちいちサーバーを挟まないため、通信のやり取り回数が少なく効率が良い。
また、サーバーの動作がシンプルになっているため、クライアント側で注文のやり方の仕様が変えやすくなる。

## メッセージシーケンス
上記のイメージで動く場合、サーバーとクライアント間で行われるやりとりはどのようなものになるだろうか？

accept、connet後
1,サーバーは「いらっしゃいませ」とクライアント送信。
2,サーバーは「メニューのリスト」をクライアントに送信。
3,クライアントは「注文リスト」をサーバーに送信。
4,サーバーは「注文を受け付けました」とクライアントに送信。

1と2の間にはクライアント側の操作は入らないため、同時に送信してしまってよいだろう。（いらっしゃいませの表示だけならクライアントでも十分なぐらい）
3のクライアントは注文のリストをサーバーに送信するがこれは必要になる。ピザのメニューは時期によって変わることがある。
4の「注文を受け付けました」の送信も必須である。クライアント側はこれを待たないとそのままプログラムを終了していいのかわからないからだ。

行われる通信は３回のため、３つのメッセージシーケンスを考えれば良い。

### 注記
クライアント側にメニュー情報をもたせておいて、時期によってメニューが変わったらクライアント側のデータをアップデートする方式にすれば
メッセージシーケンスは２つ送ればいいだけになるが、今回はメニューもサーバー側から送られてくるものとする。

## 含まれるべきメッセージ
### 1,サーバーからクライアントに注文を送る場合
メニューと価格を送る必要がある
```
product_id,menu,price,stock_quantity
```
のcsvファイルのようなプロトコルでいいだろう。

### 2,クライアントの注文リストをサーバーに送信
通信の方法には２パターンある。

これのポイントはクライアント側がどれぐらいたくさんの注文をするかをサーバー側は知らないということである。
そのため、注文の開始の合図と終了の合図がメッセージに必要になる。
例えば次のような例だ
```
START
1,3
3,5
6,2
END
```
左の数字がproduct_idで右の数字が注文個数となる。
これで１番の商品を３個、３番の商品を５個、６番の商品を２個というふうに解釈するようにすればよい。
STARTがきたら次の行から注文を読み取り、ENDがきたら注文を確定させれば良い。
このパターンは１回で通信を終えることができる。

もう１つのパターンは１行ごとメニューを送るループを作り、注文はこれでいいとなったら、
ENDと入力するとサーバーが次の処理に入るような作りである。

```
1,3  →　送信
3,5 →  送信
6,2 →　送信
END →　送信 // ENDを受けてサーバーは次の処理に移行していく。
```

このケースではサーバー側は状態を持つ必要がある。
サーバーが注文受付中なのか、注文を受け終わったのかを確認して処理を分岐させる必要がある。

```
while(state == ORDER){
    ソケットからデータ読む
        if 読んだデータがENDなら
            状態をDONEに変更
            break;
}
```

流れを図で書くと
```
/* Sをサーバー、Cをクライアントとする */

S → C : MENU

C → S : ITEM 1,3
S → C : OK

C → S : ITEM 3,5
S → C : OK

（繰り返し）

C → S : END

S → C : RESULT
```
こんなイメージで順々に読んでいく必要がある。
state_machineのような形でサーバーの動作を設計していくことによっても実現できる。


### 3,サーバーからクライアントへ受け付けた旨のメッセージ
これは受付が完了してたら完了のメッセージを送れば良い
これは正常受付だったら200、異常だったら300などのエラーコードを返してくるようなメッセージでよいだろう。


### 第２案とこのメッセージシーケンスの問題点
第２案は最初に送られてきたメニュー表が正しいという前提で全ての処理が行われている。
メニューの内容はともかくとして在庫数についてはどうだろうか？
クライアントが１つしかないならば、問題ないかもしれないが、同時に何人ものお客さんから注文が入るとしたらどうだろうか？

例えば、Aさんが残り１つのメニューを選ぼうとして、注文をしている途中に、Bさんからその最後の１つの注文が入ったら
Aさんは在庫切れのため、そのメニューを買うことはできなくなる。つまり、このケースではAさんの注文を受けることができない。

第２案でしっかり動くようにするならば、最後のサーバーからクライアントへの受付確認のメッセージをエラー処理のメッセージシーケンスを
パターンごとにしっかり作り込む必要がある。
また、エラーが返ってきたときには、クライアント側は再注文できないといけないため、実際にはクライアント側とサーバー側で状態を持ち
フローを制御していく必要がある。
